Bug Review / 工程问题复盘

你现在要根据本次对话、代码修改记录、终端错误、页面现象、用户描述和项目上下文，帮助我生成一篇“工程问题复盘文档”。

这不是普通开发文档，也不是最终代码说明，而是一次“AI 辅助开发后的坑点沉淀”。

目标是让我以后看到这篇文档时，能快速想起：

* 当时想做什么
* 出了什么问题
* 如何复现这个问题
* 我哪里理解错了
* 真正原因是什么
* 最后怎么修好的
* 下次遇到类似问题怎么排查
* 这次暴露了哪些需要补的工程知识点

⸻

生成位置

请将文档保存到：

docs/engineering-notes/bug-review/

如果该目录不存在，请自动创建。

文件名请使用下面格式：

序号-问题名称复盘.md

例如：

001-登录注册接口封装问题复盘.md
002-Axios错误拦截重复弹窗复盘.md
003-移动端布局局部滚动问题复盘.md
004-Nginx刷新页面404问题复盘.md

序号规则：

* 如果目录下已有复盘文档，请读取已有文件名，自动使用下一个序号。
* 如果目录为空，从 001 开始。
* 问题名称要简洁、准确，能看出这次问题的核心。
* 文件名不要太长，控制在 20 个中文字符左右。

⸻

写作要求

请使用中文，语气适合个人学习复盘，不要写成官方文档。

重点不要只写“最后代码怎么写”，而是要写清楚：

* 我为什么会错
* 错误理解在哪里
* AI 反复修改中暴露出什么问题
* 真正的工程原因是什么
* 后续如何避免再犯

如果当前上下文不足，不要编造。可以在文档中写：

待补充：这里需要结合具体代码或运行现象进一步确认。

⸻

文档结构

生成的 Markdown 文档必须包含下面这些部分。

⸻

xxx 问题复盘

1. 问题背景

说明这次是在什么项目、什么模块、什么需求下遇到的问题。

需要讲清楚：

* 项目背景
* 涉及页面 / 组件 / 接口 / 配置
* 原本想实现的功能
* 这次修改的目标是什么
* 为什么需要 AI 介入修改

⸻

2. 现象描述

用清楚的语言描述页面、接口、构建、部署或交互上出现了什么异常。

需要包含：

* 用户看到的现象是什么
* 控制台 / 终端 / Network / 页面表现有什么异常
* 预期结果是什么
* 实际结果是什么
* 是否存在偶现、必现、刷新后变化、环境差异等情况

不要只写“出错了”，要尽量写成以后能定位问题的描述。

⸻

3. 如何复现

请写出可以重新触发这个问题的复现步骤。

格式参考：

1. 启动项目
2. 打开某个页面
3. 执行某个操作
4. 观察页面 / 控制台 / 网络请求 / 构建结果
5. 出现某个错误现象

如果问题和特定条件有关，也要写清楚，例如：

* 特定账号 / 权限
* 特定路由
* 特定数据状态
* 特定接口返回
* 特定屏幕尺寸
* 特定浏览器
* 是否需要刷新页面
* 是否需要清缓存
* 是否和本地环境 / 测试环境 / 生产环境有关

如果当前无法完整复现，也请写明“不确定的复现条件”，不要编造。

⸻

4. 错误尝试

总结我一开始是怎么理解的，尝试了什么方式，为什么这些方式没有真正解决问题。

重点写清楚：

* 我以为问题出在哪里
* 实际上这个判断为什么不准确
* 哪些修改只是表面修复
* 哪些修改引入了新的问题
* AI 在反复修改中走过哪些弯路
* 哪些地方是靠“试出来的”，而不是一开始就理解清楚的

⸻

5. 核心原因

不要只说“代码写错了”，要解释背后的工程原因。

可以从下面这些方向分析，但只选择和本次问题相关的，不要泛泛而谈：

* 状态流转不清晰
* 组件职责边界不清楚
* CSS 布局约束互相冲突
* 接口类型没有提前定义
* 路由守卫和请求拦截器职责混乱
* Nginx / base 路径 / history 路由配置不一致
* 异步流程没有区分 loading、success、error、cancel 等状态
* 本地环境和线上环境配置不一致
* 数据结构理解错误
* 父子组件通信方向混乱
* 样式作用域、层级、定位、滚动容器没有提前确认
* 环境变量、代理、跨域、端口、安全组等配置没有统一确认
* 缓存、构建产物、部署目录、访问路径没有区分清楚

这一节要重点解释：

为什么会出现这个问题？
为什么之前的修改没有解决？
为什么最终方案能解决？

⸻

6. 最终解决方案

说明最终是如何修复的，包括关键代码、关键配置、关键设计调整。

需要说明：

* 最终采用了什么方案
* 为什么这个方案是正确的
* 它解决了哪些具体问题
* 有没有保留边界情况处理
* 有没有对原来的结构做职责调整
* 是否影响其他模块
* 是否需要后续继续优化

⸻

7. 关键代码 / 配置片段

只保留最重要的代码或配置，不要整篇复制。

每段代码下面解释：

* 这段代码解决了什么问题
* 为什么要这样写
* 如果不这样写会出现什么问题
* 哪些地方以后修改时需要特别小心

如果本次问题不是代码问题，而是配置 / 部署 / 环境问题，也可以放：

* Nginx 配置
* package.json scripts
* vite.config.ts
* 环境变量
* 请求路径配置
* 路由配置
* 宝塔 / 云服务器配置说明

⸻

8. 以后如何避免

请整理成 checklist。

可以参考下面格式：

- [ ] 修改前先确认数据流
- [ ] 先定义类型再写接口
- [ ] 复杂状态先画状态机
- [ ] CSS 布局先确认父容器约束
- [ ] 涉及登录鉴权时先确认 token、路由守卫、请求拦截器的职责
- [ ] 涉及部署时先确认 base、Nginx、try_files、端口、安全组
- [ ] 涉及滚动问题时先确认是哪一个容器在滚动
- [ ] 涉及接口问题时先确认请求参数、响应结构、错误拦截、业务状态码
- [ ] 涉及权限问题时先确认角色、路由、按钮显隐、接口权限是否一致

请根据本次问题生成具体 checklist，不要只复制模板。

⸻

9. 下次遇到类似问题怎么排查

请给出具体排查流程，不要只写原则。

格式参考：

1. 先确认问题是否可复现
2. 再确认问题发生在哪一层：页面 / 组件 / 状态 / 接口 / 路由 / 构建 / 部署
3. 查看控制台错误
4. 查看 Network 请求
5. 查看关键状态值
6. 缩小问题范围
7. 做最小化复现
8. 再修改代码
9. 修改后验证正常场景和异常场景

排查流程要结合本次问题，不要写成万能空话。

⸻

10. 这次需要补的知识点

列出我应该补的工程知识点，并简单解释为什么要补。

例如：

* Vue 组件通信
* Pinia 状态管理
* Axios 请求封装
* 路由守卫
* CSS 布局与滚动容器
* TypeScript 类型设计
* Nginx 部署配置
* 浏览器缓存与 HTTPS
* 前端鉴权流程
* 异步状态管理
* 文件上传流程
* 前端构建与部署路径
* 浏览器 DevTools 调试
* 移动端适配

请根据这次问题本身选择相关知识点，不要泛泛而谈。

⸻

11. 一句话总结

最后用几句话总结这次坑点，例如：

* 这次问题本质不是 xxx，而是 xxx。
* 以后遇到类似问题，不能只让 AI 反复改代码，要先确认 xxx。
* 真正应该沉淀的是 xxx。

⸻

输出要求

完成后请告诉我：

1. 新增的文档路径
2. 这篇复盘的核心主题
3. 这次最值得记住的 3 个坑点
4. 后续建议我补的 1-3 个知识点

不要只输出文档内容，请实际创建或修改对应 Markdown 文件。
